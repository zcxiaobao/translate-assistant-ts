## 上下文

### 当前状态
- 这是一个全新的项目，从零开始构建
- 项目使用 TypeScript 作为主要开发语言（项目名：translate-assistant-ts）
- 目标是在 120 分钟内完成 MVP 实现

### 约束条件
- **时间约束**：120 分钟开发时间，需要快速迭代和交付
- **技术约束**：需要集成第三方 AI API（如 OpenAI、Claude、国内大模型）
- **用户约束**：目标用户是产品经理和开发工程师，需要简单易用的界面

### 利益相关方
- **产品经理**：需要将产品需求翻译成技术语言
- **开发工程师**：需要将技术方案翻译成业务语言
- **团队协作**：减少沟通成本，提高协作效率

## 目标 / 非目标

**目标：**
- 实现产品→开发、开发→产品两个方向的 AI 翻译
- 提供简洁的用户界面（Web 或 CLI），支持翻译方向选择和内容输入
- 支持流式输出，实时展示 AI 响应
- 提示词设计能准确捕捉产品和开发两种角色的思维差异
- 提供基础文档，包括使用说明和测试用例

**非目标：**
- 不实现复杂的 UI 设计和前端交互
- 不支持翻译历史记录和会话管理
- 不支持多语言翻译（仅支持中文场景下的角色翻译）
- 不实现用户认证和权限管理
- 不进行大规模并发优化（MVP 阶段）

## 决策

### 1. 技术栈选择

**决策**：使用 Node.js + TypeScript + Express 构建后端服务

**理由**：
- TypeScript 提供类型安全，减少错误
- Express 轻量级，快速搭建 HTTP 服务
- Node.js 生态成熟，AI SDK 支持良好
- 熟悉的技术栈可以缩短开发时间

**备选方案**：
- Python + FastAPI：Python 在 AI 领域更流行，但项目已命名为 ts 版本
- Streamlit：快速搭建 UI，但定制性较差，不适合后续扩展

### 2. AI API 选择

**决策**：优先支持 OpenAI API，设计为可插拔的 AI 提供商接口

**理由**：
- OpenAI API 稳定且文档完善
- 支持流式输出（SSE）
- 可通过接口抽象，后续轻松扩展支持 Claude、国内大模型
- 使用环境变量配置 API Key，便于切换

**备选方案**：
- Claude API：同样优秀，但 OpenAI 使用更广泛
- 国内大模型（通义千问等）：可作为备选，接口设计上保持兼容

### 3. 界面类型选择

**决策**：使用简单的 Web UI（HTML + Vanilla JavaScript）

**理由**：
- Web UI 比 CLI 更直观，适合非技术背景的产品经理
- 无需引入 React/Vue 等框架，减少开发时间
- 支持流式输出的视觉呈现效果更好
- 便于后续扩展和美化

**备选方案**：
- CLI 工具：开发更快，但用户体验较差
- React/Vue SPA：功能强大，但开发时间超出 120 分钟预算

### 4. 提示词设计策略

**决策**：使用系统角色提示词 + 上下文示例（Few-shot Learning）

**理由**：
- 系统提示词明确定义两个角色的思维模式和关注点
- Few-shot 示例帮助 AI 理解期望的翻译风格
- 将产品和开发的关注点列表化，指导 AI 生成结构化输出

**关键设计点**：
- **产品→开发翻译**：
  - 系统角色：技术专家
  - 输出要求：技术实现方案、数据来源、性能要求、工作量评估
  - 补充缺失信息：主动提出技术关注点
- **开发→产品翻译**：
  - 系统角色：产品顾问
  - 输出要求：用户价值、业务影响、成本收益、增长空间
  - 补充缺失信息：主动提出业务关注点

### 5. 流式输出实现

**决策**：使用 Server-Sent Events (SSE) 实现流式输出

**理由**：
- SSE 简单易实现，无需 WebSocket 复杂性
- OpenAI API 原生支持 SSE 流式响应
- 浏览器原生支持 EventSource API
- 单向通信满足翻译场景需求

**备选方案**：
- WebSocket：双向通信，但本场景不需要客户端向服务器推送数据
- 长轮询：实现复杂且性能较差

### 6. 项目结构

**决策**：采用简单的分层架构

```
src/
  ├── services/
  │   ├── ai-provider.ts      # AI 提供商接口
  │   ├── openai-provider.ts  # OpenAI 实现
  │   └── translator.ts       # 翻译服务
  ├── prompts/
  │   ├── pm-to-dev.ts        # 产品→开发提示词
  │   └── dev-to-pm.ts        # 开发→产品提示词
  ├── routes/
  │   └── translate.ts        # 翻译路由
  ├── app.ts                  # Express 应用
  └── server.ts               # 服务器入口
public/
  └── index.html              # Web UI
```

**理由**：
- 清晰的职责划分
- 易于测试和维护
- 支持未来扩展

## 风险 / 权衡

### 1. API 调用成本
**风险**：频繁调用 AI API 可能产生较高成本
**缓解措施**：
- 使用环境变量控制模型选择（如 gpt-3.5-turbo 成本较低）
- 添加输入长度限制
- 后续可考虑添加本地缓存

### 2. AI 响应质量
**风险**：提示词设计不当可能导致翻译质量差
**缓解措施**：
- 在开发阶段预留时间调试提示词
- 提供测试用例验证翻译效果
- 支持快速修改提示词配置

### 3. 流式输出延迟
**风险**：网络延迟可能影响用户体验
**缓解措施**：
- 显示加载状态提示用户等待
- 流式输出让用户看到实时进展
- 设置合理的超时时间

### 4. 错误处理
**风险**：API 调用失败、网络错误、无效输入等
**缓解措施**：
- 捕获 AI API 错误并返回友好提示
- 前端输入验证
- 后端添加基础的错误日志

### 5. 时间压力
**风险**：120 分钟时间有限，可能无法完成所有功能
**缓解措施**：
- 优先实现核心翻译功能
- UI 从简，能用即可
- 文档使用模板快速生成
